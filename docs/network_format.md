# Формат сохранения нейронной сети

Этот документ описывает JSON-формат файлов сети для RINGS. Он рассчитан на
сторонних разработчиков, которые хотят генерировать совместимые файлы без
использования встроенного редактора.

Файл сохраняет только топологию и параметры LIF-сети: нейроны, синапсы и
визуальные группы. Текущее состояние симуляции не сохраняется: потенциалы,
счетчики спайков, летящие импульсы, положение агента, объекты мира и настройки
скорости среды будут созданы заново при запуске или загрузке.

## Общие правила

Файл должен быть валидным UTF-8 JSON без комментариев. Расширение обычно
`.json`. В Python-версии файлы лежат в `saved_networks/`, а локальный экспорт и
импорт работают с тем же форматом. Статичная версия `bug_web/` читает такие же
JSON-файлы как примеры.

Минимальная структура:

```json
{
  "format_version": 2,
  "task": "BUG",
  "neurons": [],
  "synapses": [],
  "groups": []
}
```

Рекомендуется всегда писать `format_version: 2` и явный `task`. Старые файлы без
этих полей загружаются как `BUG`, но новый генератор не должен полагаться на это.

Верхнеуровневые поля:

| поле | тип | обязательно | описание |
|---|---:|---:|---|
| `format_version` | integer | да | Текущая версия формата. Для новых файлов используйте `2`. |
| `task` | string | да | Задача/среда: `BUG`, `CartPole` или `MountainCar`. |
| `neurons` | array | да | Список нейронов. Порядок влияет только на порядок отрисовки и внутренние индексы. |
| `synapses` | array | да | Список направленных связей между нейронами. |
| `groups` | array | да | Визуальные рамки редактора. Можно оставить пустым массивом. |

При загрузке приложение сначала нормализует `task`. Неизвестное значение
считается `BUG`. Затем оно переключает среду, загружает топологию, восстанавливает
недостающие обязательные сенсоры и моторы для выбранной задачи и принудительно
исправляет типы системных нейронов.

## Нейроны

Каждый элемент `neurons` описывает один LIF-нейрон:

```json
{
  "id": "sensor_food_left",
  "label": "Food L",
  "kind": "sensor",
  "x": 60,
  "y": 60,
  "threshold": 1.0,
  "leak": 0.02,
  "v_reset": 0.0,
  "v_min": -100.0,
  "refractory": 2,
  "noise_std": 0.0
}
```

Поля нейрона:

| поле | тип | обязательно | значение по умолчанию при загрузке | описание |
|---|---:|---:|---:|---|
| `id` | string | да | нет | Уникальный идентификатор. Синапсы ссылаются на него через `from` и `to`. |
| `label` | string | нет | `id` | Подпись в редакторе. |
| `kind` | string | нет | `inter` | Тип: `sensor`, `motor` или `inter`. |
| `x` | number | нет | `0.0` | X-координата нейрона на схеме редактора. |
| `y` | number | нет | `0.0` | Y-координата нейрона на схеме редактора. |
| `threshold` | number | нет | `1.0` | Порог спайка. |
| `leak` | number | нет | `0.08` | Утечка потенциала за один шаг LIF. |
| `v_reset` | number | нет | `0.0` | Потенциал после спайка и во время refractory-периода. |
| `v_min` | number | нет | `-100.0` | Нижний предел потенциала после торможения. |
| `refractory` | integer | нет | `2` | Количество шагов симуляции после спайка, когда нейрон не интегрирует вход. |
| `noise_std` | number | нет | `0.0` | Стандартное отклонение гауссова шума. Отрицательные значения приводятся к `0`. |

Для новых файлов лучше задавать все поля явно. Это делает файл независимым от
дефолтов конкретной реализации и помогает одинаково открывать его в Python- и
статичной JS-версии.

Ограничения и особенности:

* `id` должен быть уникальным. Дубликаты в Python-версии приведут к ошибке
  загрузки, а в статичной версии будут проигнорированы.
* Системные сенсоры и моторы должны иметь точные `id` из таблиц ниже. Именно по
  ним среда подает внешние токи и считывает моторные спайки.
* Для обязательных нейронов загрузчик сохраняет координаты и числовые параметры
  из файла, но тип `kind` исправляет по `id`: `sensor_` и `lidar_` становятся
  `sensor`, `motor_` становится `motor`.
* Если обязательного сенсора или мотора нет в файле, приложение добавит его
  автоматически с дефолтными параметрами и координатами.
* Максимальная емкость сети в текущей реализации - `512` нейронов.

## Синапсы

Каждый элемент `synapses` описывает одну направленную связь:

```json
{
  "from": "sensor_food_left",
  "to": "motor_left",
  "weight": 1.0,
  "delay": 5
}
```

Поля синапса:

| поле | тип | обязательно | значение по умолчанию при загрузке | описание |
|---|---:|---:|---:|---|
| `from` | string | да | нет | `id` нейрона-источника. |
| `to` | string | да | нет | `id` нейрона-получателя. |
| `weight` | number | нет | `1.0` | Вес импульса. Положительный вес возбуждает, отрицательный тормозит. |
| `delay` | integer | нет | `1` | Задержка в шагах LIF-симуляции. Значение ограничивается диапазоном `1..64`. |

Один шаг LIF равен `SIM_DT = 0.01` секунды симуляционного времени, поэтому
`delay: 5` означает доставку импульса примерно через `0.05` секунды
симуляционного времени.

Связь с `weight: 0` технически не имеет эффекта и не попадает в экспорт
приложения. Если несколько записей описывают одну пару `from -> to`, фактически
останется последнее значение в матрице весов, поэтому генератору лучше не
создавать дубликаты. Синапсы с несуществующими `from` или `to` пропускаются.

## Визуальные группы

`groups` не участвуют в симуляции. Это прямоугольные рамки редактора для
документирования схемы:

```json
{
  "id": "g-food",
  "x": 40,
  "y": 30,
  "w": 180,
  "h": 140,
  "label": "Food sensors",
  "color": "#38ffb3",
  "comment": "Signals from food receptors"
}
```

Поля группы:

| поле | тип | обязательно | значение по умолчанию при загрузке | описание |
|---|---:|---:|---:|---|
| `id` | string | нет | `gN` | Идентификатор группы. |
| `x` | number | нет | `0` | X-координата рамки. |
| `y` | number | нет | `0` | Y-координата рамки. |
| `w` | number | нет | `100` | Ширина рамки. |
| `h` | number | нет | `80` | Высота рамки. |
| `label` | string | нет | `""` | Подпись рамки. |
| `color` | string | нет | `"#3a5cff"` | Цвет рамки в CSS-формате. |
| `comment` | string | нет | `""` | Комментарий, показываемый в редакторе как подсказка. |

Генератор может оставить `groups: []`, если визуальная разметка не нужна.

## Различия между задачами

Главное различие задач - набор обязательных системных нейронов. Эти нейроны
являются интерфейсом между сетью и средой. Внешний генератор может добавлять
любые `inter`-нейроны и синапсы, но должен использовать правильные `id`
сенсоров и моторов для выбранного `task`.

### `BUG`

`BUG` - оригинальная 2D-среда с едой, угрозами, голодом, усталостью и лидарами.

Обязательные сенсоры:

| id | label | смысл |
|---|---|---|
| `sensor_food_left` | `Food L` | Сила сигнала еды слева. |
| `sensor_food_right` | `Food R` | Сила сигнала еды справа. |
| `sensor_threat_left` | `Threat L` | Сила сигнала угрозы слева. |
| `sensor_threat_right` | `Threat R` | Сила сигнала угрозы справа. |
| `sensor_hunger` | `Hunger` | Текущий голод агента. |
| `sensor_fatigue` | `Fatigue` | Текущая усталость агента. |
| `lidar_0` | `Lidar 1` | Луч препятствий 1. |
| `lidar_1` | `Lidar 2` | Луч препятствий 2. |
| `lidar_2` | `Lidar 3` | Луч препятствий 3. |
| `lidar_3` | `Lidar 4` | Луч препятствий 4. |
| `lidar_4` | `Lidar 5` | Луч препятствий 5. |

Обязательные моторы:

| id | label | действие |
|---|---|---|
| `motor_forward` | `Forward` | Движение вперед. |
| `motor_backward` | `Back` | Движение назад. |
| `motor_left` | `Turn L` | Поворот влево. |
| `motor_right` | `Turn R` | Поворот вправо. |

Рекомендуемые параметры системных нейронов для генератора:

| тип | `threshold` | `leak` | `v_reset` | `v_min` | `refractory` | `noise_std` |
|---|---:|---:|---:|---:|---:|---:|
| BUG-сенсор | `1.0` | `0.02` | `0.0` | `-100.0` | `2` | `0.0` |
| BUG-мотор | `1.0` | `0.12` | `0.0` | `-100.0` | `3` | `0.0` |

### `CartPole`

`CartPole` использует `CartPole-v1`. Для каждого наблюдения создается пара
сенсоров: положительный компонент и отрицательный компонент. Если сырое
наблюдение положительное, активнее `*_pos`; если отрицательное, активнее
`*_neg`.

Обязательные сенсоры:

| id | label | наблюдение |
|---|---|---|
| `sensor_cart_position_pos` | `Cart X +` | Положительная позиция тележки. |
| `sensor_cart_position_neg` | `Cart X -` | Отрицательная позиция тележки. |
| `sensor_cart_velocity_pos` | `Cart V +` | Положительная скорость тележки. |
| `sensor_cart_velocity_neg` | `Cart V -` | Отрицательная скорость тележки. |
| `sensor_pole_angle_pos` | `Pole angle +` | Положительный угол шеста. |
| `sensor_pole_angle_neg` | `Pole angle -` | Отрицательный угол шеста. |
| `sensor_pole_angular_velocity_pos` | `Pole ang V +` | Положительная угловая скорость шеста. |
| `sensor_pole_angular_velocity_neg` | `Pole ang V -` | Отрицательная угловая скорость шеста. |
| `sensor_reward` | `Reward` | Нормированный сигнал качества эпизода. |

Обязательные моторы:

| id | label | действие |
|---|---|---|
| `motor_left` | `Left` | Действие `left`. |
| `motor_right` | `Right` | Действие `right`. |

Рекомендуемые параметры системных нейронов:

| тип | `threshold` | `leak` | `v_reset` | `v_min` | `refractory` | `noise_std` |
|---|---:|---:|---:|---:|---:|---:|
| CartPole-сенсор | `1.0` | `0.02` | `0.0` | `-100.0` | `0` | `0.0` |
| CartPole-мотор | `1.0` | `0.12` | `0.0` | `-100.0` | `3` | `0.0` |

Для `CartPole` `sensor_reward` использует режим `survival_up`: сигнал растет от
`0` к `1` по мере приближения к лимиту эпизода в `500` шагов.

### `MountainCar`

`MountainCar` использует `MountainCar-v0`. Как и в `CartPole`, каждое наблюдение
разбивается на положительный и отрицательный сенсор. Для позиции используется
специальная нормализация относительно диапазона среды, поэтому генератору
достаточно использовать правильные `id`.

Обязательные сенсоры:

| id | label | наблюдение |
|---|---|---|
| `sensor_position_pos` | `Position +` | Позиция в положительном направлении нормировки. |
| `sensor_position_neg` | `Position -` | Позиция в отрицательном направлении нормировки. |
| `sensor_velocity_pos` | `Velocity +` | Положительная скорость. |
| `sensor_velocity_neg` | `Velocity -` | Отрицательная скорость. |
| `sensor_reward` | `Reward` | Нормированный сигнал оставшегося бюджета времени. |

Обязательные моторы:

| id | label | действие |
|---|---|---|
| `motor_left` | `Left` | Толкать влево. |
| `motor_coast` | `Coast` | Ничего не делать. |
| `motor_right` | `Right` | Толкать вправо. |

Рекомендуемые параметры системных нейронов:

| тип | `threshold` | `leak` | `v_reset` | `v_min` | `refractory` | `noise_std` |
|---|---:|---:|---:|---:|---:|---:|
| MountainCar-сенсор наблюдения | `1.0` | `0.02` | `0.0` | `-100.0` | `0` | `0.0` |
| `sensor_reward` | `10.0` | `0.02` | `0.0` | `-100.0` | `0` | `0.0` |
| MountainCar-мотор | `1.0` | `0.12` | `0.0` | `-100.0` | `3` | `0.0` |

Для `MountainCar` `sensor_reward` использует режим `time_budget_down`: сигнал
стартует около `1` и падает к `0` к лимиту эпизода в `200` шагов. Если цель
достигнута, среда сообщает успех и сбрасывает эпизод.

## Выбор действия по моторам

Сеть не сохраняет отдельную таблицу действий. Действие определяется спайками
моторных нейронов текущей задачи:

* `BUG`: активные моторы напрямую применяются к движению агента; несколько
  моторов могут быть активны одновременно.
* `CartPole` и `MountainCar`: среда собирает активные `motor_*` в список
  допустимых действий и выбирает одно действие для шага Gymnasium-совместимой
  среды. Если ни один мотор не активен, используется дефолтное действие среды
  (`coast` для `MountainCar`).

Поэтому генератору важно подключать свои внутренние нейроны именно к моторным
`id` выбранной задачи. Синапс к мотору другой задачи не будет использоваться
после переключения `task`, потому что такой мотор не входит в текущий список
действий.

## Рекомендуемый алгоритм генерации

1. Выберите `task`.
2. Создайте верхнеуровневый объект с `format_version: 2`, `task`, пустыми
   массивами `neurons`, `synapses`, `groups`.
3. Добавьте все обязательные сенсоры и моторы выбранной задачи. Это не строго
   обязательно для загрузчика, но делает файл самодостаточным и предсказуемым.
4. Добавьте пользовательские `inter`-нейроны с уникальными `id`. Практичный
   префикс: `n_`, например `n_food_gate`.
5. Добавьте синапсы. Убедитесь, что каждый `from` и `to` существует в
   `neurons`, `delay` находится в диапазоне `1..64`, а дубликатов пар
   `from -> to` нет.
6. При необходимости добавьте визуальные `groups`.
7. Провалидируйте JSON: уникальность `id`, наличие массивов, корректность
   `task`, отсутствие ссылок на несуществующие нейроны.

## Manifest для статичной версии

Формат самой сети не зависит от `manifest.json`. Manifest нужен только
статичной папке `bug_web/saved_networks/`, чтобы показать список примеров в UI:

```json
{
  "files": [
    "start.json",
    "simple.json"
  ],
  "groups": {
    "BUG": [
      "start.json",
      "simple.json"
    ],
    "CartPole": [],
    "MountainCar": []
  }
}
```

`files` содержит имена JSON-файлов рядом с manifest. `groups` необязателен, но
рекомендуется: он группирует примеры по `task` в диалоге загрузки. Каждый файл в
группе все равно должен иметь собственное поле `task`.
